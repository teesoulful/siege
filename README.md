# Squad Ops — strategy database

A local web app that turns YouTube setup/defense breakdowns into a structured,
per-site strategy database you can pull up mid-match. Runs entirely client-side —
no accounts, no backend, data lives in `data/` as JSON and gets bundled into
`dist/data.js` for the app to load.

## Layout

```
squad_ops_prototype_v4.html   the app (single HTML file, vanilla JS)
build_data.py                 combines data/*.json -> dist/data.js
scripts/fetch_transcripts.py  yt-dlp + VTT-to-clean-text pipeline for new videos
data/
  PARSING_SPEC.md              schema + ground rules handed to whoever/whatever parses a transcript
  operators.json               full operator roster: side + gadget + utility count (side is wiki-verified, counts still pending)
  maps/<map-slug>.json         bomb sites + room/callout names for a map, wiki-verified
  strategies/<map-slug>_<side>.json   structured strategy entries parsed from one transcript
  raw_transcripts/             cleaned [mm:ss]-timestamped transcript text, one file per video
dist/data.js                  generated — do not edit by hand
<MAP NAME>/                   ad-hoc source material (e.g. Bank's original transcript + blueprint images)
```

## How a map gets added

1. **Get the transcript.** Add the video to `scripts/fetch_transcripts.py`'s
   `VIDEOS` list (video id, map name, side) and run
   `python3 scripts/fetch_transcripts.py <video_id>` — this downloads
   YouTube's auto-captions via `yt-dlp` and cleans them into deduped
   `[mm:ss]`-tagged text in `data/raw_transcripts/`. (First-time setup on a new
   machine: `pip3 install --user yt-dlp certifi`, and export
   `SSL_CERT_FILE=$(python3 -c "import certifi; print(certifi.where())")` if
   you hit an SSL verify error — the python.org installer ships without a
   populated cert bundle.)

2. **Parse the transcript into `data/strategies/<map-slug>_<side>.json`.**
   Read `data/PARSING_SPEC.md` for the exact schema and ground rules (no
   invented content, every item cites a timestamp, flag ASR ambiguity instead
   of guessing an operator name). `data/strategies/bank.json` is the worked
   example. This step was mostly done by parallel subagents for the current
   dataset — see "Lessons from the first big batch" below before doing this at
   scale again.

3. **Verify site/callout names against the wiki**, and write
   `data/maps/<map-slug>.json` with the official bomb site pairs and room
   names (`verified: true` once sourced — Liquipedia's Rainbow Six wiki and
   r6coaching.com both worked well; the Fandom wiki was returning HTTP 402 as
   of this writing). The transcript will use informal creator callouts — those
   stay in `data/strategies/*.json` since that's what a squad actually says
   out loud, but `data/maps/*.json` should carry canonical names.

4. **Cross-check operator names against `data/operators.json`'s `sides` map**
   (wiki-verified) — this catches the single most common and costly transcript
   error: recommending an operator that's actually on the wrong side, or that
   doesn't exist at all (see the "Legion" incident below). Exact gadget/utility
   *counts* are not yet populated (`utility: {}`) — no reliable single source
   was found; would need per-operator wiki page lookups.

5. **Run the build and refresh:**
   ```
   python3 build_data.py
   ```
   then reload the browser tab. No server restart needed — `dist/data.js` is
   just a static file the HTML `<script>`-tags in.

## Lessons from the first big batch (16 maps, 26 transcripts, ~1600 strategy items)

- **Cross-reference every operator name against a real, current roster before
  trusting it — including your own manual parses.** Auto-captions mangle
  operator names constantly, and it's tempting to "reconstruct" a plausible
  one from context. The worst recurring case: "Legion" doesn't exist as an R6
  operator, but it sounds enough like **Lesion** (whose gadget actually is
  goo/Gu mines, reinforcing the false confidence) that it got hand-typed by a
  human (me, on the original Bank file) and independently regenerated by six
  different parsing subagents on six other maps, all making the same mistake
  the same way. Some subagents went further and asserted "Legion AND Lesion
  are both Attack-only" as justification for refusing to name a defender —
  confidently wrong is worse than uncertain. Fixed with a global
  `Legion → Lesion` pass plus manual cleanup of the explanatory notes that
  became self-contradictory after the blind regex swap.
- **A wiki-research pass can complete a lot of real work and still self-report
  as having failed**, if it hits a session/budget limit mid-run — check
  `data/operators.json` / `data/maps/*.json` on disk before assuming a "hit
  session limit" notification means nothing landed. In this batch, ~21 of 26
  transcript-parsing agents fully succeeded despite a limit cutting off the
  batch; only the stragglers needed a targeted retry.
- **When two transcripts cover the same map from different angles** (a
  dedicated-map defense video vs. a general-attack-principles video, or two
  differently-scoped defense videos), the site names/scope they use for "the
  same" physical site can differ. The site-picker UI keys off `workingName`
  string equality between a map's Defense and Attack files — mismatched names
  silently produce a "no data yet" panel even when data exists under a
  different name for the same site. Worth a scripted post-check (compare
  Defense vs Attack site-name sets per map) any time two files land for one
  map.

## Data integrity rules this project follows

- **No invented strategy content.** Every entry in `data/strategies/*.json`
  traces back to a real timestamp in a real transcript. The original prototype
  had made-up placeholder tactics for Chalet/Clubhouse — those were deleted
  rather than shipped, because confidently-worded fake advice is worse than an
  honest "no data yet" message during a live match.
- **`verified: false`** on a map or operator file means it hasn't been
  cross-checked against the R6 Wiki yet — treat exact numbers in it as
  unconfirmed until that flag flips to `true`.
- The app shows "No verified [side] setup data yet for this site" instead of
  silently falling back to generic advice when data is missing for a
  map/side/site combination — this is intentional, not a bug to hide.

## Running the app

```
python3 -m http.server 8642 --directory "/Users/tee/Documents/SIEGE"
```
then open `http://localhost:8642/squad_ops_prototype_v4.html`. Opening the HTML
file directly (`file://`) also works since data loads via a `<script src>` tag,
not `fetch()`.

## Current coverage

16 maps, ~1600 structured strategy items, sourced from 26 YouTube transcripts:

| Map | Defense | Attack |
|---|---|---|
| Bank, Border, Calypso Casino, Chalet, Clubhouse, Coastline, Kafe Dostoyevsky, Nighthaven Labs, Oregon | yes | yes |
| Consulate, Fortress, Kanal, Outback, Skyscraper, Theme Park, Villa | yes | no source video yet |

Plus `data/strategies/general_attack_principles.json` — map-agnostic attacking
principles (drone discipline, breach economy, roles, comms) shown as a
fallback wherever a specific map/site has no dedicated Attack data yet.

`data/maps/*.json` is wiki-verified for 15/16 maps; **Calypso Casino's is
`verified: false`** — it's a newer map and no source had confirmed official
callouts yet, so it holds clearly-flagged placeholder zone names pending a
re-pass.
